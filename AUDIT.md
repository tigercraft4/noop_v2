**1. [privacy-egress] android/app/src/main/java/com/noop/data/DeviceRegistry.kt:86** `CONFIRMED` · _kotlin-data+ingest_
- deleteDeviceData() omits the device-keyed sleepStateSample (v15) and liveSession (v16) tables, so a "delete all data" leaves raw biometric samples on disk.
- **Cenário:** A privacy-first user taps "delete data" for a WHOOP in Devices. DeviceRegistry.deleteDeviceData(id) runs the fixed DELETE set in DeviceRegistryDao (deleteHrFor…deleteDayOwnershipFor, lines 65-81), which enumerates hrSample, rrInterval, spo2Sample, skinTempSample, respSample, gravitySample, stepSample, ppgHrSample, event, battery, dailyMetric, sleepSession, journal, workout, appleDaily, metricSeries, dayOwnership. **5. [correctness] Packages/OuraProtocol/Sources/OuraProtocol/OuraDriver.swift:278** `CONFIRMED` · _swift-OuraProtocol+Design_
- 0x71 green_ibi_and_amp records are decoded with the 0x60 decoder, which has a different layout, producing wrong IBIs that feed HRV scoring.
- **Cenário:** EventTag 0x71 (greenIbiAmp) is Tier A and ingest() routes it to OuraDecoders.decodeIBIAmplitude (OuraDriver.swift:277-278), the same decoder used for 0x60. Per OURA_PROTOCOL.md 6.1 vs 6.2 these layouts DIFFER: 0x60 is '6 absolute IBI+amplitude pairs, shift exponent = last-byte low nibble [3:0]', but 0x71 is '5 IBI DELTAS + 6 amplitudes, shift from byte19 bits [2:0]'. decodeIBIAmplitude (Decoders.swift:62-79) always reads 6 IBIs as 11-bit ABSOLUTE values interleaved with 7-bit amps and takes shift = b[last] & 0x0F (4 bits). Decoding a real green-LED record therefore: (a) emits 6 IBIs where only 5 exist, fabricating a phantom R-R from amplitude bits; (b) treats delta-encoded values as absolute intervals; (c) applies an amplitude shift up to 15 instead of the correct max 7. These bogus IBIs are emitted as OuraEvent.ibi and, per 6.9, NOOP reconstructs RMSSD/HRV from the IBI stream — so wrong physiological R-R intervals silently corrupt HRV scoring, violating the honest-data invariant. (The .ibi tag 0x44 at OuraDriver.swift:274-276 is routed through the same 0x60 decoder with no 0x71/0x44-specific layout in the spec, same risk.)
- **Fix:** Give 0x71 (and 0x44 if it has its own layout) a dedicated decoder: read 5 IBI deltas + 6 amplitudes with the [2:0] shift, and accumulate the deltas into absolute IBIs; or drop 0x71 to a gated/undecoded state until a fixture confirms the byte layout, rather than reusing decodeIBIAmplitude.

**2. [concurrency] android/app/src/main/java/com/noop/ble/WhoopBleClient.kt:4735** `CONFIRMED` · _kotlin-ble+protocol_
- Check-then-set on @Volatile backfillDraining has a lost-wakeup race that can strand the last queued offload frame and stall the history sync.
- **Cenário:** onInbound() runs synchronously on the GATT binder thread (onCharacteristicChanged at 3390/3399, no handler.post), so routeBackfillFrame runs on the binder thread while the drain loop runs on ioScope (Dispatchers.IO). Interleaving: (1) the IO drainer's `backfillFrameQueue.poll()` returns null and it exits the while loop, heading to `finally`; (2) before it runs `backfillDraining = false`, the binder thread executes routeBackfillFrame: `backfillFrameQueue.add(frame)` then `if (backfillDraining) return` sees the flag still true and returns WITHOUT launching a drainer; (3) the drainer's finally sets backfillDraining=false. - **Fix:** Close the wakeup gap: in the drain coroutine's finally, after `backfillDraining = false`, re-check `if (backfillFrameQueue.isNotEmpty())` and relaunch the drain; or serialize enqueue+flag+drain under a single lock/Mutex so an add can never observe a stale draining=true while the queue is non-empty and no drainer is active.

**5. [data-loss] Strand/App/LiveSessionRunner.swift:244** `PLAUSIBLE` · _swift-App+System_
- The start-of-session and end-of-session row writes run as two independent unstructured Tasks with no ordering guarantee, so the start write (endTs=nil, zeroed totals) can land after the end write and clobber the finished session.
- **Cenário:** `start()` calls `persist(row(endTs: nil, ...))` which spawns an unstructured `Task` (line 244) that awaits `repo.storeHandle()` then `upsertLiveSession`. `end()` later calls `persist(final)` spawning a second such Task. The upsert is idempotent on (deviceId, startTs), so both target the SAME row. If the store is still opening at session start (fresh launch) and the user starts then ends a session before the store handle resolves — or the two actor writes otherwise resume out of order — the start Task's write executes last and overwrites the completed row with endTs=nil, inBandSec=0, belowSec=0, aboveSec=0.
